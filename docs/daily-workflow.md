# Daily Workflow System

This document covers the daily loop that powers the Brain personal assistant system: how the EOD generates tomorrow's plan, how the morning review confirms it, and how the two routines create a self-sustaining daily rhythm.

---

## The Daily Loop

The system operates on a continuous daily cycle:

```
 End of day  Run /eod (walk away while it processes)
             ├── Phase 1: Gather data (transcripts, email, Slack, calendar)
             ├── Phase 2: Sync + hygiene (dedup, task sync, archiving)
             ├── Phase 3: Time tracking (if configured)
             ├── Phase 4: Daily note (permanent record)
             └── Phase 5: Generate Tomorrow's Today.md
                  |
                  v
  Morning    Read Today.md (pre-built plan)
  Morning    Run /morning (interactive 3-5 min review)
  Morning    Deep work begins
                  |
                 ...  Day happens (meetings, tasks, Slack, email)
                  |
                  v
 End of day  Run /eod again
```

The key insight: **tomorrow's plan is built before you close the laptop**. The morning review takes 3-5 minutes of confirming a pre-built plan, not 30 minutes of figuring out what to do.

## Today.md: The Daily Plan

`Inbox/Today.md` is an ephemeral file, overwritten every night by EOD Phase 5. It contains everything you need to start the day.

### Conventions

#### Source Tags
Every work task includes an invisible HTML comment linking back to its source:
```
- [ ] **[Client A]** Task description <!-- src:Inbox/ClientA.md|fingerprint text -->
```

The fingerprint is a 30-40 character substring from the task text in the source file, unique enough to identify the exact task via substring match. This enables the morning review to trace tasks back to their origin and the carry-forward system to track which tasks persist across days.

#### Meeting Tags
Calendar meetings use a different tag:
```
- [ ] 1:00 Meeting Title (Client, duration) <!-- type:meeting -->
```
Meetings don't carry forward (they're either attended or not).

#### Carry-Forward Tags
Items that weren't completed yesterday and are re-selected for today:
```
- [ ] **[Client A]** Overdue task <!-- src:Inbox/ClientA.md|fingerprint --><!-- carried:2 -->
```
The number increments each day. Items carried for 3+ days surface as a concern during the morning review.

#### Obsidian Wiki-Links
References to vault files use `[[wiki-links]]` so you can click through directly in Obsidian:
```
**Recent activity:** [[2026-03-14 - Client B Weekly Sync]]
```

## /morning: Interactive Review

The `/morning` command is designed for minimal friction. On a clean day, it completes in 3 taps.

### Flow

```
Step 1: Load Today.md
        ├── Missing? Offer to generate
        ├── Stale date? Offer to regenerate
        └── Valid? Proceed
            |
            v
Step 2: Summary (8-10 lines)
        "Good morning. Tuesday, March 18.
         3 meetings today -- first at 1:00, last at 2:30
         Full deep work block protected
         Top priority: [Client A] API integration
         No imminent deadlines"
            |
            v
Step 3: Client Review (batch)
        Present ALL clients at once:
        ├── [Client A]: 2 tasks, ~2 hrs, looks clean
        ├── [Client B]: 1 task + meeting, ⚠ staging unverified
        └── [Client C]: 1 task, ~30 min
        Capacity: ~4.5 hrs needed, ~5 hrs available
            |
            v
        AskUserQuestion: "Any clients need adjustments?"
        ├── "All good" ──────────> Step 4
        ├── "[Client B]" ────────> Follow-up questions
        └── "Other" (free text) ─> Parse and apply
            |
            v
Step 4: Goal Check
        ├── All current? Skip (no output)
        └── Stale goals detected?
            AskUserQuestion: "Refresh now?"
            ├── "Skip" ──────> Step 5
            └── "Quick refresh" > Collect + update
            |
            v
Step 5: Calendar Time Blocks
        Propose blocks:
        | 8:05-8:20  | LinkedIn |
        | 8:20-8:35  | Team priorities |
        | 8:35-12:00 | Deep Work 1: [Client A], [Client B] |
        | 2:30-5:30  | Deep Work 2: [Client C], admin |
            |
        AskUserQuestion: "Create these blocks?"
        ├── "Create all" ────> Create on Google Calendar
        ├── "Adjust first" ──> Modify then create
        └── "Skip blocks" ──> No calendar events
            |
            v
Step 6: Send-off
        "Plan locked. 4 time blocks created.
         First meeting at 1:00 ([Contact Name] sync).
         [Client A] API integration starts now in Deep Work 1."
```

### Design Principles

- **Every decision uses `AskUserQuestion`** with selectable options. No free-typing required for common choices.
- **Batch then select.** Show everything at once, then ask which items need changes. Don't walk through clients one by one.
- **Flags, not nagging.** Stale goals get mentioned once with a skip option. No blocking, no guilt.
- **Overcommitment guard.** If estimated work exceeds available deep work time, force a prioritization conversation before creating blocks.
- **Time block granularity.** Group related small tasks into 30-90 minute blocks. Don't create 15-minute blocks for everything.

## EOD Pipeline Architecture

The default `/eod` should run in one Claude session. Claude Code now supports long-context sessions, so most users do not need to split the workflow anymore. Keep the phased version as an advanced fallback for unusually heavy days or unattended scheduled automation.

### Why Phases?

Historically, EOD was split because long daily closeout sessions could overflow the model context on busy days. With long-context Claude Code sessions, that workaround is no longer the default recommendation. Keep phased EOD only when a specific user's workflow is still too large for one run, or when you need extra resilience for unattended scheduled jobs.

### Phase Architecture

Default architecture:
```
You type: /eod
    │
    ▼
One Claude session
    ├── Gather: transcripts, email, Slack, calendar, brain dump
    ├── Route: write items to inbox files and manifest
    ├── Sync: dedup, cleanup, task manager sync
    ├── Note: write Work/Daily/YYYY-MM-DD.md
    └── Plan: write Inbox/Today.md for tomorrow
```

Advanced fallback:
```
You type: /eod
    │
    ▼
Phased /eod
    ├── Gather
    ├── Sync
    ├── Time (optional)
    ├── Note
    └── Plan
```

For either approach:
- Write important state to disk as you go
- Route items immediately instead of batching them in memory
- Skip sections for tools the user does not use

### The Manifest File

The manifest (`/tmp/eod-manifest-TODAY.md`) is the audit trail:

```markdown
# EOD Manifest -- 2026-03-18

## Items

| # | Item | Client | Type | Source | Routed To | Status |
|---|------|--------|------|--------|-----------|--------|
| 1 | Review PR #234 | [Client A] | action-owner | Transcript: [Contact] call | Inbox/ClientA.md | ✓ |
| 2 | Follow up: [Team Member] to send mockups | [Client B] | followup | Transcript: [Contact] call | Inbox/ClientB.md | ✓ |
| 3 | Reply re: timeline | [Client C] | email-response | Email: [Contact]@... | Inbox/ClientC.md | ✓ |
| 4 | Staging deploy status? | [Client B] | research | Slack: DM | Inbox/ClientB.md | ✓ |
```

Phase 1 creates the manifest. Phase 2 reads it for deduplication. Phase 5 reads it for priority ranking (today's items rank higher than older inbox items).

### Failure Handling

- If Phase 1 fails, later phases still run with whatever data was written before the failure
- If Phase 2 fails, time tracking and daily note still generate
- You can ask Claude to re-run a specific phase if one fails
- Each phase is independent: partial data is better than no data

## Advanced: Scheduled Automation

For power users who want the EOD to run on a schedule (e.g., 11:30 PM weekdays) without manual intervention. This requires terminal setup and is completely optional. Most users just run `/eod` manually.

See `examples/scripts/` in the setup repository for:
- `eod-runner.sh` -- Shell orchestrator that runs each phase as a separate Claude CLI invocation
- `eod-cron.sh` -- Cron wrapper with version pinning, lockfiles, and macOS Gatekeeper handling
- `com.brain.eod-runner.plist` -- macOS `launchd` config for scheduling

Key considerations for unattended execution:
- **Version pinning**: Pin a specific Claude CLI version to avoid macOS TCC permission dialogs from auto-updates
- **Lockfiles**: Prevent overlapping runs
- **Notifications**: macOS `osascript` for local, Slack DM for remote failure alerts

## The Monolithic Alternative

The default recommendation is to keep `/eod` as one command:

```markdown
# End of Day

## Setup
Source .env, verify date, create manifest.

## Section 1: Call Transcripts
Pull call transcripts, save, extract action items, route.

## Section 2: Tomorrow's Calendar
Fetch events, format schedule.

## Section 3: Email
Surface items needing response, route.

## Section 4: Slack
Check all workspaces, route.

## Section 5: Time Tracking
Pull Rize data, classify, generate relabel checklist.

## Section 6: Verify + Sync
Dedup, task manager sync, hygiene.

## Section 7: Daily Note
Generate summary.

## Section 8: Today.md
Generate tomorrow's plan.
```

Start with this. Split into phases when you notice context overflow (items getting lost, sections being skipped, or the AI apologizing for running out of space).

## Key Patterns

### Route-As-You-Go

Every extracted item gets routed to its destination file immediately, not batched for later. This prevents items from being lost if:
- The AI's context gets compressed (long sessions)
- A phase fails partway through
- The session is interrupted

### Atomic Writes for iCloud

If your vault syncs via cloud storage, use Python read-modify-write instead of the AI's built-in file editor:

```python
python3 << 'PYEOF'
with open("Inbox/ClientA.md", "r") as f:
    content = f.read()
# Insert new task under Open Tasks
content = content.replace("### Open Tasks\n", "### Open Tasks\n- [ ] New task -- *from source*\n")
with open("Inbox/ClientA.md", "w") as f:
    f.write(content)
PYEOF
```

The built-in file editor reads the file, then writes it as a separate operation. If iCloud syncs a change in between, the write overwrites the sync. The Python script does both in a single process.

### Deduplication

Before adding any item, check if it already exists:
- Same task text (fuzzy match) in the same client file
- Same Slack message timestamp in the processed state file
- Same email subject + sender from the same day

When a duplicate is found, update the source note ("*also discussed 3/18*") instead of creating a new entry.

### Carry-Forward Decay

Tasks in Today.md have a carry count. This creates natural pressure:
- Day 1: Task appears fresh
- Day 2: `<!-- carried:1 -->` (noted in morning review)
- Day 3: `<!-- carried:2 -->` (flagged as concern)
- Day 4+: If it keeps getting re-selected but never done, the morning review surfaces it explicitly

Tasks that aren't re-selected simply stay in the client inbox file. They don't carry forward; they drop off Today.md and wait for a future prioritization cycle.
